# 04 — Camada de IA (OpenRouter + AI SDK v6)

Parte do planejamento do FisioPilates — ver 00-visao-geral.md

Este documento cobre a camada de inteligência do produto: integração técnica com o OpenRouter via Vercel AI SDK v6, os três casos de uso (análise de evolução, análise postural com visão, chat assistente com RAG), o schema estruturado do relatório, o builder de dossiê pseudonimizado, os guardrails de saúde em camadas, o controle de custos e o tratamento de erros/rate limits em ambiente serverless.

Convenção de idioma (ver 00-visao-geral.md): código, nomes de tabela/coluna, campos de schema e identificadores permanecem em inglês; textos de UI e conteúdo gerado saem em pt-BR.

## 0. Decisões-resumo (TL;DR)

| Decisão | Escolha | Justificativa |
|---|---|---|
| Modelo principal | `anthropic/claude-sonnet-5` via OpenRouter, **pinado por env** (`OPENROUTER_MODEL`) | Sonnet mais recente; visão + file input; contexto de 1M; nunca usar alias `latest` em produção |
| Modelo barato (tarefas simples) | `anthropic/claude-haiku-4.5` | Títulos de conversa, sumarização de histórico e de sessões antigas, multi-query do RAG |
| Fallback pinado | `anthropic/claude-sonnet-4.6` | Via parâmetro `models` do OpenRouter (fallback automático na mesma requisição) |
| Integração | **Vercel AI SDK v6 (`ai` ^6) + `@openrouter/ai-sdk-provider` ^2** | Streaming, `useChat`, tool-calling e `generateObject` prontos; troca de provedor trivial |
| Relatórios de evolução | `generateObject` (JSON estruturado com Zod), **síncrono** (`maxDuration = 300`) | Renderização consistente, cache por `input_hash`, auditoria; sem streaming de tokens e sem fila |
| Chat com RAG | `streamText` + tool-calling (`buscar_conhecimento`, `buscar_ficha_aluno`, `buscar_web`), `stopWhen: stepCountIs(5)` | O modelo decide quando buscar e formula queries melhores que o embedding da mensagem crua |
| Fotos posturais | Signed URL do Supabase Storage → content part `image` | Sonnet 5 aceita imagem via OpenRouter; evita base64 gigante na função serverless |
| Privacidade no prompt | Dossiê **pseudonimizado** (sem nome completo/CPF/contato) + `provider.data_collection: 'deny'` | Fronteira LGPD: dado sensível nunca vaza para retenção/treino do provedor |

Duas decisões estruturais que valem para tudo abaixo:

- **Relatório é síncrono, não tem fila.** A geração roda dentro da própria rota (`maxDuration = 300`, Fluid Compute no plano Pro). A UI acompanha por **polling de status** (`generating` → `ready`/`failed`), não por streaming de tokens. O QStash é usado **apenas** para ingestão de KB (ver 03-rag.md), nunca para relatórios.
- **Chat é streaming.** Só o chat usa `streamText` + `toUIMessageStreamResponse()`; ali o streaming melhora a percepção de latência e o `useChat` cuida do estado.

## 1. Integração técnica

### 1.1 Por que AI SDK v6 + provider OpenRouter (e não fetch direto)

Usamos o **Vercel AI SDK v6** (`ai` ^6) com o **`@openrouter/ai-sdk-provider` ^2**. O fetch direto contra a API OpenAI-compatible do OpenRouter só valeria a pena se precisássemos de algum recurso não exposto pelo provider — hoje `extraBody`/`providerOptions` cobrem tudo (usage accounting, provider routing, fallbacks). Manter o AI SDK também deixa a porta aberta para trocar de provedor (ex.: API Anthropic direta) sem reescrever a UI.

| Critério | AI SDK + provider | fetch direto (OpenAI-compatible) |
|---|---|---|
| Streaming SSE para o front | `toUIMessageStreamResponse()` pronto | parse manual de SSE |
| Hook de chat no React | `useChat` (estado, retry, abort, anexos) | tudo manual |
| Tool-calling multi-step | `tools` + `stopWhen: stepCountIs(n)` | loop manual de `tool_use` |
| Saída estruturada | `generateObject` + Zod | parse/validação manual |
| Troca/fallback de modelo | 1 linha (`extraBody.models`) | 1 linha (empate) |
| Recursos específicos OpenRouter | `extraBody` cobre todos | acesso total (empate) |

```bash
npm i ai @openrouter/ai-sdk-provider zod
```

### 1.2 `lib/ai/client.ts` — factory e tabela de modelos

Os IDs de modelo **nunca** são literais espalhados pelo código: vêm de env (`OPENROUTER_MODEL`, `OPENROUTER_MODEL_FALLBACK`, `OPENROUTER_MODEL_CHEAP`), validados em `lib/env.ts` com zod. Isso permite trocar de modelo/versão sem redeploy de código e evita `latest` em produção.

```ts
// lib/ai/client.ts
import { createOpenRouter } from '@openrouter/ai-sdk-provider';
import { env } from '@/lib/env';

export const openrouter = createOpenRouter({
  apiKey: env.OPENROUTER_API_KEY,
  // headers de atribuição no ranking do OpenRouter (opcionais)
  headers: {
    'HTTP-Referer': 'https://app.fisiopilates.com.br',
    'X-Title': 'FisioPilates',
  },
});

// IDs pinados via env — jamais literais nem alias `latest`.
export const MODELS = {
  analysis: env.OPENROUTER_MODEL,        // relatórios (qualidade máxima) — anthropic/claude-sonnet-5
  chat: env.OPENROUTER_MODEL,            // chat técnico — anthropic/claude-sonnet-5
  cheap: env.OPENROUTER_MODEL_CHEAP,     // títulos, resumos, multi-query — anthropic/claude-haiku-4.5
} as const;

// Fallback automático do OpenRouter (via extraBody.models) — anthropic/claude-sonnet-4.6
export const FALLBACK_MODELS = [env.OPENROUTER_MODEL_FALLBACK] as const;

// Opções aplicadas a TODA chamada de geração/streaming (LGPD + custo real).
export const OPENROUTER_EXTRA_BODY = {
  models: FALLBACK_MODELS,                // fallback na mesma requisição se o provedor primário falhar
  usage: { include: true },              // retorna custo real em USD na resposta
  provider: { data_collection: 'deny' }, // só provedores que NÃO retêm/treinam com os dados
} as const;
```

Valores esperados em produção (ver 06-infra-deploy.md para a lista completa de envs):

```dotenv
OPENROUTER_MODEL=anthropic/claude-sonnet-5
OPENROUTER_MODEL_FALLBACK=anthropic/claude-sonnet-4.6
OPENROUTER_MODEL_CHEAP=anthropic/claude-haiku-4.5
```

Três parâmetros do `extraBody` são **obrigatórios** em toda chamada e por isso centralizados em `OPENROUTER_EXTRA_BODY`:

- **`provider.data_collection: 'deny'`** — restringe o roteamento a provedores que não retêm nem treinam com os dados enviados. É requisito de LGPD (dado de saúde), não opcional.
- **`usage: { include: true }`** — faz o OpenRouter devolver o **custo real em USD** (não só contagem de tokens) na resposta, alimentando `ai_usage_log` no `onFinish`.
- **`models: FALLBACK_MODELS`** — fallback nativo do OpenRouter: se o provedor primário estiver saturado/fora, ele tenta `claude-sonnet-4.6` na mesma requisição, sem custo de código.

### 1.4 Pontos serverless / Vercel

- **`runtime = 'nodejs'`** em todas as rotas de IA (não Edge): precisamos do client Supabase server-side com service role e da assinatura de signed URLs.
- **`maxDuration`**: `120` no chat; `300` no relatório. Fluid Compute (plano Pro) permite folga suficiente; o relatório roda de ponta a ponta dentro desse teto — não há job em background.
- Toda rota, na ordem: valida sessão Supabase Auth → resolve `tenant_id` (nunca vindo do client) → **checa cota** (`assertQuota`, seção 4) → só então chama o modelo.

Esqueleto do chat (streaming, C6):

```ts
// app/api/ai/chat/route.ts
import { streamText, convertToModelMessages, stepCountIs } from 'ai';
import { openrouter, MODELS, OPENROUTER_EXTRA_BODY } from '@/lib/ai/client';
import { chatSystemPrompt } from '@/lib/ai/prompts/chat-assistente';
import { buildChatTools } from '@/lib/ai/tools';
import { requireTenant } from '@/server/auth';
import { assertQuota, logUsage } from '@/lib/ai/usage';

export const runtime = 'nodejs';
export const maxDuration = 120;

export async function POST(req: Request) {
  const { messages, chatId } = await req.json();
  const { tenantId, userId } = await requireTenant();     // auth + RLS
  await assertQuota(tenantId);                             // 402 amigável se estourou

  const result = streamText({
    model: openrouter(MODELS.chat, { extraBody: OPENROUTER_EXTRA_BODY }),
    system: chatSystemPrompt(),                            // estável → prompt caching Anthropic
    messages: convertToModelMessages(messages),
    tools: buildChatTools({ tenantId, userId }),           // buscar_conhecimento/ficha_aluno/web
    stopWhen: stepCountIs(5),
    onFinish: ({ usage }) => logUsage({ tenantId, feature: 'chat', usage, chatId }),
  });

  // consumeStream garante onFinish (gravação de uso) mesmo se o cliente desconectar.
  result.consumeStream();
  return result.toUIMessageStreamResponse();
}
```

| Modelo (ID exato, via env) | Input /1M | Output /1M | Contexto | Modalidades |
|---|---|---|---|---|
| `anthropic/claude-sonnet-5` | ~$3.00 | ~$15.00 | 1M | texto, imagem, arquivo |
| `anthropic/claude-sonnet-4.6` | ~$3.00 | ~$15.00 | 1M | texto |
| `anthropic/claude-haiku-4.5` | ~$1.00 | ~$5.00 | 200K | texto, imagem |

Observações:

- **Orçamos a preço cheio ($3/$15)** para não depender de preço introdutório. Qualquer estimativa neste documento assume esse patamar.
- **Prompt caching da Anthropic funciona através do OpenRouter** (breakpoints `cache_control`), com ~90% de desconto no prefixo repetido. Aplicar no system prompt do chat (estável entre requisições) e no bloco de conhecimento RAG reaproveitado.
- O ID nunca é alias `latest`: é pinado por env (`OPENROUTER_MODEL`). Deriva de preço/ID no OpenRouter é mitigada por env + fallback automático + monitoração de crédito (seção 5).

## 2. Casos de uso e design de prompts

### 2.1 Análise de evolução do aluno (relatório periódico)

O diferencial de produto. Botão "Gerar análise com IA" na aba Evolução da ficha do aluno → `POST /api/ai/analyses` com `{ studentId, period: { from, to } }` → backend monta o dossiê pseudonimizado → `generateObject` → persiste em `ai_reports` → UI faz polling do status e renderiza as seções.

**É síncrono (C8):** a rota `maxDuration = 300` gera e persiste dentro da própria requisição; não há streaming de tokens nem fila. A UI dispara o POST, recebe o `id` do relatório com status `generating`, e faz **polling** em `GET /api/ai/analyses/[id]` até `ready` ou `failed`.

**Dados de entrada (montados server-side, nunca vindos do client)** — detalhados no builder da seção 2.4:

1. **Ficha de avaliação/anamnese** — queixa principal, diagnóstico médico informado (se houver), histórico, objetivos, restrições/contraindicações, condições/patologias.
2. **Medidas/avaliações físicas** — série temporal (perimetria, flexibilidade, força, dor EVA…), como tabela compacta `data | métrica | valor`.
3. **Histórico de sessões do período** — data, exercícios, aparelho, carga/molas, séries/reps, observações. Se > 40 sessões, sumarizar as antigas com Haiku e mandar a íntegra só das últimas ~20.
4. **Documentos relevantes** — `extracted_text` de exames/laudos (extração feita no upload — ver 02-banco-de-dados.md e 03-rag.md; nunca extraído na hora da análise).
5. **Contexto RAG** — multi-query (Haiku gera 3–5 queries a partir de queixa + objetivos + padrões), dedupe, ~10–12 chunks da base de conhecimento com metadados de fonte.

**System prompt (pt-BR).** O bloco de guardrails (seção 3) é concatenado a partir de `lib/ai/guardrails.ts`; abaixo o texto completo do system da análise:

```
Você é um assistente de análise clínica para fisioterapeutas que utilizam o método
Pilates. Sua função é APOIAR a profissional, analisando dados registrados por ela
e produzindo um relatório técnico de evolução.

REGRAS INEGOCIÁVEIS:
1. Você NÃO faz diagnóstico. Nunca afirme que o aluno "tem" uma condição clínica;
   no máximo descreva achados e padrões observados nos dados registrados.
2. Você NÃO prescreve tratamento, medicamento ou conduta. Você pode SUGERIR pontos
   de atenção e possibilidades de progressão de exercícios, sempre condicionadas
   ao julgamento clínico da fisioterapeuta responsável.
3. Baseie TODA afirmação nos dados fornecidos. Se um dado não existe, escreva
   "não há registro" — nunca invente valores, datas ou evoluções.
4. Use o CONTEXTO DE CONHECIMENTO fornecido como referência técnica; se ele não
   cobrir o tema, diga que a base de conhecimento não cobre o assunto.
5. Escreva em português do Brasil, tom profissional e claro, tratando a leitora
   como colega fisioterapeuta (pode usar termos técnicos).
6. Termine sempre reforçando que o relatório é um apoio e que a conclusão clínica
   é da profissional.
```

O dossiê entra na mensagem `user` em blocos demarcados por tags (`<ficha>`, `<medidas>`, `<sessoes>`, `<documentos>`, `<conhecimento>`), com a instrução final: *"Gere o relatório de evolução do período X a Y."*

**Schema de saída — `generateObject` + Zod.** JSON estruturado (não markdown livre) permite renderização consistente, seções colapsáveis, cache por `input_hash` e auditoria. O campo `aviso` é um `z.literal(...)` que **força o disclaimer** em todo relatório (guardrail de schema — seção 3).

```ts
// lib/ai/schemas/relatorio.ts
import { z } from 'zod';

export const relatorioSchema = z.object({
  resumo_executivo: z.string().describe('3-5 frases, linguagem clara'),
  evolucao_pilates: z.object({
    progressao_exercicios: z.string(),   // markdown permitido dentro dos campos
    carga_e_complexidade: z.string(),
    aderencia: z.string(),               // frequência, faltas, regularidade
  }),
  evolucao_corporal: z.object({
    medidas_e_tendencias: z.string(),
    dor_e_queixas: z.string(),
    observacoes_posturais: z.string().nullable(), // preenchido só quando há fotos (2.2)
  }),
  pontos_de_atencao: z.array(z.string()),
  sugestoes_para_proximas_sessoes: z
    .array(z.string())
    .describe('Sugestões condicionais ao julgamento clínico, nunca prescrições'),
  dados_faltantes: z.array(z.string()), // transparência do que não havia registro
  aviso: z.literal(
    'Este relatório é um apoio gerado por IA e não substitui a avaliação da fisioterapeuta responsável.',
  ),
});

export type Relatorio = z.infer<typeof relatorioSchema>;
```

Esqueleto da rota (síncrona, com cache por `input_hash` e retry/backoff):

```ts
// app/api/ai/analyses/route.ts
import { generateObject } from 'ai';
import { openrouter, MODELS, OPENROUTER_EXTRA_BODY } from '@/lib/ai/client';
import { relatorioSchema } from '@/lib/ai/schemas/relatorio';
import { buildDossie, hashDossie } from '@/lib/ai/dossie';
import { analiseSystemPrompt } from '@/lib/ai/prompts/analise-evolucao';
import { requireTenant } from '@/server/auth';
import { assertQuota, logUsage, withRetry } from '@/lib/ai/usage';

export const runtime = 'nodejs';
export const maxDuration = 300;

export async function POST(req: Request) {
  const { studentId, period } = await req.json();
  const { tenantId } = await requireTenant();
  await assertQuota(tenantId);

  const dossie = await buildDossie({ tenantId, studentId, period }); // pseudonimizado (2.4)
  const inputHash = hashDossie(dossie);

  // Cache por input_hash (C10): se nada mudou, devolve o relatório existente sem chamar o modelo.
  const cached = await findReport({ tenantId, studentId, period, inputHash });
  if (cached) return Response.json({ id: cached.id, status: cached.status, cached: true });

  // Reserva a linha em status 'generating' com a constraint única — barra duplo clique (seção 5.6).
  const row = await createGeneratingReport({ tenantId, studentId, period, inputHash, dossie });

  try {
    const { object, usage } = await withRetry(() =>
      generateObject({
        model: openrouter(MODELS.analysis, { extraBody: OPENROUTER_EXTRA_BODY }),
        schema: relatorioSchema,
        system: analiseSystemPrompt(),
        prompt: dossie.promptUser,
        maxOutputTokens: 4000,
        abortSignal: AbortSignal.timeout(90_000),
      }),
    );
    await finalizeReport(row.id, { structured: object, status: 'draft' }); // nasce RASCUNHO (3)
    await logUsage({ tenantId, feature: 'analysis', usage, reportId: row.id });
    return Response.json({ id: row.id, status: 'draft' });
  } catch (err) {
    await failReport(row.id);
    throw mapAiError(err); // → mensagem pt-BR (seção 5.4)
  }
}
```

Observações de persistência (ver 02-banco-de-dados.md para o DDL de `ai_reports`):

- `structured jsonb` é a fonte de verdade (saída do `generateObject`); `content_md` é render opcional para export.
- `input_hash = sha256(dossiê serializado)`; unique `(tenant_id, student_id, report_type, period, input_hash)` dá idempotência e cache (C10).
- `input_snapshot` guarda o dossiê usado (pseudonimizado) para auditoria de "sobre quais dados este relatório foi feito".
- Status: `generating` → `draft` (nasce rascunho) → `final` (após aprovação humana). `failed` é re-disparável.

### 2.2 Análise corporal/postural (visão)

Sonnet 5 aceita imagens via OpenRouter. No MVP a análise postural é uma **seção opcional do relatório de evolução** (preenche `evolucao_corporal.observacoes_posturais`), acionada quando há fotos no período — não um endpoint próprio.

**Como enviar fotos do Supabase Storage:**

1. Fotos ficam em bucket **privado** (`student-documents`), com RLS por tenant e prefixo `{tenant_id}/`.
2. Na hora da análise o backend gera **signed URLs curtas** — `createSignedUrl(path, 300)` (5 min) — e as passa como content parts `image` no AI SDK; o provedor baixa a imagem pela URL. Isso evita base64 gigante inflando payload e memória da função. Fallback (só se a signed URL falhar): baixar server-side e mandar base64.
3. **Pré-processamento no upload** (ver 03-rag.md / Fase 4): redimensionar para ~1568px no lado maior com `sharp` — acima disso só aumenta token de imagem sem ganho (~1,5–2k tokens/foto). **Limite de 6 fotos por análise** (frente/costas/perfis, antes/depois).
4. Emparelhar cada foto com metadados na mensagem: `"Foto 1: vista anterior, 12/03/2026"`.

**Bloco de prompt acrescentado ao system da análise** (`lib/ai/prompts/analise-postural.ts`):

```
Ao analisar fotos posturais:
- Descreva apenas o observável: alinhamentos, assimetrias aparentes, posição de
  ombros/pelve/joelhos, comparação entre datas.
- Ressalte as limitações do meio (ângulo da câmera, roupa, iluminação podem
  distorcer a percepção).
- NÃO atribua causa clínica a achados visuais (ex.: não conclua "escoliose";
  escreva "assimetria de ombros aparente, a confirmar em avaliação física").
- Se as fotos não permitirem análise confiável, diga isso explicitamente.
```

**LGPD:** foto de corpo é dado sensível. Além do `provider.data_collection: 'deny'` já em `OPENROUTER_EXTRA_BODY`, o consentimento do aluno é registrado no cadastro (bloqueante), e o prompt **nunca** inclui nome completo/CPF — apenas o identificador pseudonimizado do builder (2.4).

Montagem das content parts (esquema):

```ts
const fotos = await getPeriodPhotos({ tenantId, studentId, period, limit: 6 });
const imageParts = await Promise.all(
  fotos.map(async (f, i) => ({
    type: 'image' as const,
    image: await signedUrl(f.path, 300),   // 5 min
    // legenda vai no texto: `Foto ${i + 1}: ${f.view}, ${formatDate(f.taken_at)}`
  })),
);
```

### 2.3 Chat assistente com RAG (tool-calling)

A profissional tira dúvidas técnicas em linguagem natural, com fontes citadas. Usa `streamText` + `useChat`; persistência no formato `UIMessage` do AI SDK (ver `chat_messages.parts jsonb` em 02-banco-de-dados.md — preserva tool calls para replay/hidratação).

**RAG por tool-calling (C6), não injeção fixa:** nem toda mensagem precisa de busca ("reformule esse texto") e o modelo formula queries melhores que o embedding da mensagem crua. Três tools, com `stopWhen: stepCountIs(5)` limitando o loop:

```ts
// lib/ai/tools.ts
import { tool } from 'ai';
import { z } from 'zod';
import { ragSearch, tavilySearch } from '@/lib/ai/rag';
import { getStudentSnapshot } from '@/server/students';

export function buildChatTools({ tenantId, userId }: { tenantId: string; userId: string }) {
  return {
    buscar_conhecimento: tool({
      description:
        'Busca na base de conhecimento de Pilates/fisioterapia da plataforma. Use SEMPRE que a pergunta envolver técnica, indicação/contraindicação de exercício, patologia ou princípios do método.',
      inputSchema: z.object({
        consulta: z.string().describe('Pergunta reformulada para busca semântica'),
      }),
      execute: async ({ consulta }) => ragSearch(consulta, { tenantId, k: 6 }),
    }),

    buscar_ficha_aluno: tool({
      // permite "como está a evolução da Maria?"
      description:
        'Busca dados de um aluno da própria profissional (ficha, últimas sessões, medidas).',
      inputSchema: z.object({ nomeOuId: z.string() }),
      execute: async ({ nomeOuId }) => getStudentSnapshot({ tenantId, userId, nomeOuId }), // RLS isola
    }),

    buscar_web: tool({
      description:
        'Busca na web (Tavily) em fontes técnicas. Use APENAS quando buscar_conhecimento não retornar conteúdo suficiente, e informe à usuária que a fonte é externa.',
      inputSchema: z.object({ consulta: z.string() }),
      execute: async ({ consulta }) => tavilySearch(consulta),
    }),
  };
}
```

- `buscar_conhecimento` chama o `ragSearch` híbrido (pgvector + FTS `portuguese` com RRF — ver 03-rag.md); resultados voltam com metadados de fonte (`titulo`, `pagina`) para o modelo citar (`[KB-1]`, "segundo *Apostila X*, p. 12").
- `buscar_ficha_aluno` roda sob RLS (o snapshot só enxerga alunos do próprio tenant); é o que responde "como está a evolução da Maria?" com dado real e só do tenant.
- `buscar_web` é o gatilho de fallback externo (Tavily, allowlist e cache — ver 03-rag.md); a resposta sinaliza a fonte como `[WEB-n]`/externa.

**System prompt do chat** (`lib/ai/prompts/chat-assistente.ts`) — **estável e idêntico entre requisições**, marcado com cache breakpoint (prompt caching Anthropic via OpenRouter) para pagar ~10% do input nas mensagens seguintes:

```
Você é o assistente técnico da plataforma, conversando com uma fisioterapeuta
formada que dá aulas de Pilates. Responda dúvidas técnicas com base prioritária
na base de conhecimento (ferramenta buscar_conhecimento). Cite a fonte quando
usar material da base. Se usar busca na web, deixe claro que a fonte é externa
e menos confiável. Não diagnostique nem prescreva para casos de alunos —
ofereça raciocínio técnico e referências para apoiar a decisão DELA.
Escreva em português do Brasil. Seja direto; use listas quando ajudar.
Se a pergunta fugir de fisioterapia/Pilates/gestão do estúdio, recuse com
gentileza e redirecione.
```

**Janela de contexto:** enviar as últimas ~30 mensagens; se passar, sumarizar as antigas com Haiku (`MODELS.cheap`) e injetar como resumo. **Título automático** da conversa gerado com Haiku após a 1ª troca.

### 2.4 Builder de dossiê pseudonimizado (`lib/ai/dossie.ts`)

Regra de arquitetura, não convenção: **o prompt nunca recebe identificadores diretos** (nome completo, CPF, telefone, e-mail, endereço). O builder monta o contexto server-side sob RLS e substitui a identidade por um rótulo estável (primeiro nome ou `Aluno(a)`), coerente com a fronteira LGPD (ver 07-lgpd-seguranca.md).

Responsabilidades:

1. Buscar, sob RLS/tenant, ficha + condições + sessões do período + medidas + `extracted_text` de documentos relevantes.
2. **Pseudonimizar**: remover/omitir nome completo, CPF, contatos e endereço; manter só o rótulo de exibição e dados clínicos.
3. **Truncar/sumarizar**: se > 40 sessões, resumir as antigas com Haiku e manter a íntegra das últimas ~20 (teto de tokens).
4. Rodar **multi-query RAG**: Haiku gera 3–5 queries a partir de queixa/objetivos/padrões → `ragSearch` de cada uma → dedupe → ~10–12 chunks com fonte.
5. Emitir `promptUser` (blocos com tags) + `snapshot` (para `input_snapshot`) + `hashDossie(snapshot)` para o `input_hash`.

```ts
// lib/ai/dossie.ts (contrato)
export type Dossie = {
  promptUser: string;   // <ficha>…</ficha><medidas>…</medidas>… + instrução final
  snapshot: unknown;    // versão serializável, já pseudonimizada → gravada em input_snapshot
};

export async function buildDossie(args: {
  tenantId: string;
  studentId: string;
  period: { from: string; to: string };
}): Promise<Dossie> {
  // 1. carrega dados sob RLS  2. pseudonimiza  3. trunca/sumariza (Haiku)
  // 4. multi-query RAG (Haiku)  5. monta blocos com tags e a instrução final
}

// input_hash: hash determinístico do snapshot (ordem estável de chaves).
export function hashDossie(d: Dossie): string {
  return sha256(stableStringify(d.snapshot));
}
```

O `input_hash` deriva do `snapshot` **pseudonimizado e estável** — reordenações ou campos identificadores não devem mudar o hash, senão o cache (C10) nunca acerta.

## 3. Guardrails de saúde (em camadas)

O risco regulatório/clínico (interpretação como diagnóstico, responsabilidade COFFITO) é tratado em quatro camadas independentes — nenhuma sozinha basta:

1. **System prompt (camada 1).** Regras inegociáveis de não-diagnóstico / não-prescrição, ancoragem nos dados ("se não existe, escreva 'não há registro'"), "não sei" explícito quando a base não cobre o tema. Vivem em `lib/ai/guardrails.ts` e são **concatenadas a todos os prompts** (análise, postural, chat).

2. **Schema de saída (camada 2).** O campo `aviso` com `z.literal(...)` **força o disclaimer** em todo relatório — o `generateObject` só valida se o texto exato estiver presente. As sugestões são tipadas/descritas como "condicionais ao julgamento clínico", nunca prescrições.

3. **Humano no circuito + UI (camada 3).**
   - **Relatório nasce `draft` (rascunho).** A profissional **revisa e aprova** antes de marcar como `final`/compartilhar. Isso é essencial para responsabilidade profissional (COFFITO): a IA propõe, a fisioterapeuta assina.
   - **Banner fixo** em todo conteúdo gerado: *"Conteúdo gerado por IA como apoio à profissional. Não substitui avaliação, diagnóstico ou conduta fisioterapêutica."*
   - No chat, **subtítulo permanente**: *"Assistente de apoio técnico — as decisões clínicas são suas."*

4. **Escopo do produto (camada 4).** A IA **nunca é exposta ao aluno final** no MVP — só à profissional. Isso reduz drasticamente o risco regulatório.

Transversal — **LGPD no prompt**: minimização de dados (dossiê pseudonimizado — 2.4), `provider.data_collection: 'deny'` no OpenRouter, consentimento registrado e bloqueante, e `ai_usage_log` **sem conteúdo clínico** (só métricas: `kind`, modelo, tokens, custo, `generation_id` em `metadata`).

## 4. Controle de custos

### 4.1 Estimativas por operação (a preço cheio $3/$15)

| Operação | Input típico | Output típico | Custo estimado |
|---|---|---|---|
| Relatório de evolução (sem fotos) | ~12k tok (system 1,5k + ficha 2k + sessões 4k + medidas 1k + RAG 3k) | ~2,5k tok | ~$0,075 |
| Relatório com 4–6 fotos | +8–12k tok | ~3k tok | ~$0,12 |
| Mensagem de chat (com 1 busca RAG) | ~6k tok (cai p/ ~2k efetivos com cache) | ~0,8k tok | ~$0,03 |
| Título/resumo (Haiku) | ~1k | ~0,1k | <$0,002 |

Ordem de grandeza: uma profissional ativa (30 alunos, 1 relatório/aluno/mês + ~200 mensagens de chat) ≈ **$6–10/mês** de IA — confortável para precificar um plano futuro.

### 4.2 Mecanismos

1. **`ai_usage_log`** `(id, tenant_id, user_id, kind, model, input_tokens, output_tokens, cost_usd, metadata, created_at)` — DDL em 02-banco-de-dados.md; `kind` ∈ `('report','chat','embedding','multi_query')`. Preenchida no `onFinish` a partir de `usage: { include: true }` (custo real em USD, não estimado). O `generation_id` do OpenRouter é guardado em `metadata` e permite auditar via `GET /api/v1/generation`.

2. **Cota mensal por tenant.** Campo `ai_monthly_limit_usd` em `tenants` (default $20.00 — ver DDL em 02-banco-de-dados.md). `assertQuota(tenantId)` soma `cost_usd` do mês corrente e, se estourou, devolve erro amigável ("Você atingiu o limite de uso de IA deste mês") com HTTP 402 próprio. Estrutura pronta para virar diferenciação de planos no billing futuro.

   ```ts
   // lib/ai/usage.ts (contrato)
   export async function assertQuota(tenantId: string): Promise<void> {
     const [used, limit] = await Promise.all([
       sumMonthlyCost(tenantId),        // soma cost_usd do mês corrente em ai_usage_log
       getTenantLimit(tenantId),        // ai_monthly_limit_usd
     ]);
     if (used >= limit) throw new QuotaExceededError(); // → 402 pt-BR na rota
   }
   ```

3. **Cache de relatório por `input_hash` (C10).** `input_hash = sha256(dossiê)`; se nada mudou no aluno desde o último relatório do mesmo período, retorna o cacheado **sem chamar o modelo**. Botão "Regerar" com confirmação para forçar. A constraint única `(tenant_id, student_id, report_type, period, input_hash)` garante idempotência.

4. **Prompt caching Anthropic** (chat: system estável + bloco de contexto) — ~90% de desconto no prefixo repetido.

5. **Modelo certo por tarefa (C13).** Haiku (`MODELS.cheap`) para títulos, sumarização de histórico e multi-query do RAG; Sonnet só onde a qualidade importa (relatório e resposta do chat).

6. **Tetos técnicos.** `maxOutputTokens` 4k (relatório) / ~1,5k (chat); truncagem do dossiê (sumarizar sessões antigas); `k` fixo no RAG.

7. **Painel "Uso de IA neste mês"** em Configurações (`GET /api/ai/usage`) — transparência para a profissional e antecipação do billing.

## 5. Erros e rate limits do OpenRouter em serverless

### 5.1 Erros relevantes

`429` (rate limit — global por conta, não por key), `402` (crédito insuficiente na **conta OpenRouter** — problema nosso), `408`/timeout do provedor upstream, `502/503` (provedor fora). Créditos monitorados via `GET /api/v1/key` (`limit_remaining`).

### 5.2 Fallback nativo do OpenRouter (primeira linha)

`extraBody.models: ['anthropic/claude-sonnet-4.6']` (já em `OPENROUTER_EXTRA_BODY`): se o provedor primário falhar/saturar, o OpenRouter tenta o próximo **na mesma requisição**, custo zero de código.

### 5.3 Retry com backoff (só não-streaming)

Para o **relatório** (`generateObject`, não-streaming), `withRetry` faz 2 tentativas com backoff exponencial (1s / 4s + jitter) apenas para `429`/`5xx`/erros de rede. Orçar os retries dentro do `maxDuration = 300` e do `AbortSignal.timeout(90_000)`.

No **chat** (streaming) não dá para re-tentar no meio do stream: erro no início vira evento de erro que o `useChat` expõe em `onError`, com botão "Tentar novamente" (`reload()`); erro no meio marca a mensagem parcial como interrompida.

### 5.4 Mapeamento de erros para a usuária (pt-BR, sem jargão)

| Condição | Mensagem à usuária | Ação interna |
|---|---|---|
| `429` / `503` | "O serviço de IA está sobrecarregado. Tente novamente em instantes." | log; contabiliza taxa de erro |
| `402` (crédito OpenRouter) | "IA temporariamente indisponível." | **alerta interno de severidade alta** (Sentry/Slack) — é problema nosso, não dela |
| Cota do tenant (seção 4) | "Você atingiu o limite de uso de IA deste mês." | HTTP 402 próprio |
| Timeout/abort | "A geração demorou demais. Tente novamente." | marca relatório `failed` (re-disparável) |

### 5.5 Timeout / abort e gravação de uso

`AbortSignal.timeout(90_000)` nas chamadas. No chat, `result.consumeStream()` no servidor garante que `onFinish` (gravação em `ai_usage_log`) rode **mesmo se o cliente desconectar** — não perdemos contabilização de custo.

### 5.6 Idempotência do relatório (anti-cobrança dupla)

A linha em `ai_reports` é criada em status `generating` **antes** da chamada ao modelo, protegida pela constraint única `(tenant_id, student_id, report_type, period, input_hash)`. Duplo clique não gera nem cobra 2x: a segunda tentativa cai no registro existente (ou no cache C10). Job que falhou fica `failed` e pode ser re-disparado.

### 5.7 Observabilidade e créditos

- Logar `generation_id` do OpenRouter em `ai_usage_log` (auditoria de custo via `GET /api/v1/generation`).
- Alerta por **taxa de erro** (> 5% em 10 min).
- **Auto-top-up** de créditos no OpenRouter + alerta quando `limit_remaining < X`. Em serverless não há daemon: a checagem é feita dentro do próprio `assertQuota()` com **cache de 10 min** para não consultar a cada request.
